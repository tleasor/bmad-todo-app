## Challenges

1. **Tests passed but the product was broken.** This is the headline. Across Epics 1–4, the full test suite (260+ unit, growing E2E, 98%+ coverage) was green. Issue A only manifested in real-browser conditions the test environment didn't reproduce — `prefers-reduced-motion: reduce` zeros `animation-duration`, and some browser engines skip `animationend` on zero-duration animations entirely. Issue B was not asserted at all — no test enforced visual parity to `ux-design-preview.html`. Passing CI ≠ working product.

2. **Animation-coupled mutation was wrong from spec, not introduced by drift.** Story 3.2's *original* AC #5 specified the `animationend`-driven mutation. The pattern was greenlit in planning, implemented to spec, reviewed against spec, and shipped. The defect was specified, not regressed into. This is the deepest learning of Epic 5 — adversarial AC review for cross-cutting concerns (browser quirks, accessibility settings, motion preferences) belongs *before* implementation, not after defects surface.

3. **The `<Index>` vs `<For>` choice carried a latent defect across two epics.** SolidJS `<Index>` keys list children by position. Once any state-bearing closure (e.g., `createSignal(false)` for `isLeaving`) is set on a slot, subsequent occupants of that slot inherit the state. Epic 3's animation-and-fade pattern was the trigger; the defect lay dormant in Epics 1–4 because no test exercised second-delete-after-first. The right default for state-bearing rows is `<For>` keyed by identity; `<Index>` is for stateless rendering only.

4. **Style fragmentation across three languages produced silent drift.** Per-story styling decisions accumulated divergence from the design preview because no story owned cross-cutting visual fidelity. UnoCSS-in-JSX competed with per-component CSS competed with `reset.css`. Each story shipped its own ACs cleanly; the *aggregate* visual didn't. Story 5.2 was the first story whose AC explicitly was "matches the preview." That AC should have existed in Epic 1.

5. **Stale Docker artifact caused a sharp false-negative review.** After Stories 5.1 and 5.2 both reported success and tests passed, Tommy ran `docker compose up` (without `--build`) and observed the *old* SPA bundle from May 2 — leading to "the UI still looks the same as before" and several wasted turns of investigation chasing a non-existent code defect. The Dockerfile builds inside the image; source changes do not propagate without `--build` (or volume mounts in dev). Distinguishing runtime artifact (dev / build / container) from code defect is itself a debugging skill.

6. **Subagents reported "done" without runtime verification of the deploy artifact.** Both Story 5.1 and 5.2 dev subagents accurately reported "tests pass" against the local test runner. Neither verified that the deployed artifact (the container Tommy was actually looking at) reflected their changes. Definition-of-done for stories that affect the deployable bundle should include a runtime-artifact verification step — at minimum, "rebuild the container and confirm a sentinel value from the change is present."

7. **Story 5.3 (Visual Regression Checkpoint) was deferred and then skipped.** The story was specified in the Sprint Change Proposal as recommended-but-not-required, with the explicit rationale "if maintenance mode is the plan, manual visual checklist may suffice." Tommy made the call to skip after 5.2 closed. Whether this was right depends on future change velocity. The current consolidation onto a single source of truth (Story 5.2) means drift can't recur silently *as long as `ux-design-preview.html` is treated as contract*, but no automated check enforces that. See Insight 4 and Action Items below.

## Key Insights

1. **AC contracts must include real-use verification, not just behavioral assertions.** Epic 4's retro called this out as "AC enumeration discipline" — the same root cause produced Epic 5's existence. The lesson hardens here: ACs for UI behaviors must include the conditions under which the behavior must hold (motion preference, browser engine, viewport, theme, AT). An AC that says "delete works" is incomplete; an AC that says "delete works with `prefers-reduced-motion: reduce` and across browser engines that skip `animationend` on zero-duration animations" is what would have caught Issue A at planning time. The former passes tests and ships defects; the latter forces the test design that catches them.

2. **Default to identity-keyed lists for state-bearing rows.** `<For>` over `<Index>` should be the established pattern in this codebase. The architecture doc now codifies this (Story 5.1's edit). Future PRs that introduce `<Index>` for state-bearing children should be flagged in review.

3. **Visual fidelity to a design preview is itself an AC dimension that must be owned by a specific story or check.** "Every story shipped its ACs" is not the same as "the aggregate matches the spec." Either a per-story matches-the-preview AC or a single owner-story (the role Story 5.2 played) is required. If neither exists, drift accumulates silently. Story 5.2 closed the gap once; the open question is how to keep it closed.

4. **Skipping Story 5.3 means visual-parity enforcement is a manual ritual.** With three styling languages collapsed into one source of truth, the *capacity* for silent drift is reduced — adding a CSS rule to the wrong place is now a more visible code-review artifact than it was before. But the only enforcement mechanism is human review against `ux-design-preview.html`, which depends on a reviewer remembering to do the comparison. Whether that's enough depends on future velocity. If new visual changes are rare, manual ritual is fine. If a UX redesign or new states arrive, Story 5.3 (or equivalent) becomes worth revisiting.

5. **Runtime-artifact verification is a distinct skill from code verification.** "I changed the code, the tests pass, the change is in" is a chain that breaks at the artifact boundary. Container images, build caches, service workers, CDN edges, and stale browser tabs are all places where the deployed artifact can diverge from the source tree. A "what am I looking at?" check (sentinel value, build hash, file modification time) takes seconds and prevents the false-negative review trap. This is worth establishing as a personal habit, not as a process artifact.

6. **The Sprint Change Proposal is the right vehicle for this class of issue.** Mid-sprint discoveries that are larger than a story but smaller than a re-plan need a discrete artifact. The 2026-05-05 proposal is a clean example — it names the defects, classifies impact (Moderate), preserves audit trail, and produces a scoped, sequenced epic. Re-use the format.
