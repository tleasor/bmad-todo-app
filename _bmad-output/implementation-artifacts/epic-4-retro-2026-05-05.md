## Key Insights

1. **AC enumeration discipline is the bar to beat.** Epic 3 set "review patch volume" as the standard. Epic 4's review patches were almost universally "AC dimension N had no test." A pre-merge AC-coverage matrix (one row per AC, columns for unit / E2E / axe) is cheap and would catch most of them.

2. **Testing-only stories pay back twice.** Story 4.2 (5 tests, 0 source) found 6 contract gaps in existing code. Story 4.5 (7 tests, 0 source) found and worked around 4 cross-browser realities. When implementation is already correct, the test-writing exercise becomes a contract audit. Budget for it deliberately in future epics.

3. **Doc-vs-code drift is a real category.** UX-DR13 surfaced only because 4.5 went to verify it. Other drift items likely hide in `ux-design-specification.md`. Worth a deliberate sweep before the next focus-management story.

4. **Cross-browser surface emerges late.** Story 4.5 was the first multi-browser keyboard journey; that is when the WebKit/Chromium/Firefox differences surfaced. Future epics touching browser APIs (drag/drop, clipboard, IndexedDB) should pre-flight cross-browser before story breakdown, not after implementation.
