## Challenges

1. **Story 3.4 had six review patch items** — the highest in project history. Several were correctness bugs findable with unit tests before review: splice order bug (wrong restore positions for sequential deletes), index `-1` fallback writing a phantom task to the store, `firstDeleteAnnouncementSent` not resetting after window expiry.
2. **`navigator.platform` was deprecated in 3.3, deferred, then used again in 3.4.** Two-story deferral of a known issue. Fixed in this retro's cleanup.
3. **The 'optional' label became a parking lot.** The Docker image fix was carried as 'optional' for three epics — two of which it turned out to be already done. No tracking slot meant no one checked.

---

## Key Insights

1. **Complex stories need a pre-implementation design note.** Story 3.4 had timed state + concurrent mutations + platform detection + restore ordering. Some of the review patches were foreseeable from the design surface. A brief design note before dev begins would have surfaced them.
2. **'Optional' without a tracking slot is not a commitment and not a closure.** The Docker item was already fixed but carried as open for two epics because no one checked. Every retro action item needs a slot or a formal close.
3. **The 'keep doing better' standard applies to review patch volume.** Six patches in one story is the bar to beat in Epic 4.
