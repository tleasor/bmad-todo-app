# Deferred Work

## Deferred from: code review of 1-1-initialize-monorepo-scaffold-with-toolchain (2026-04-30)

- `bun-types` is declared in `apps/web/devDependencies` and listed in `apps/web/tsconfig.json` `"types"`, even though no SPA source consumes Bun-only globals. The architectural boundary says only `apps/api/src/env.ts` should consume `Bun.env`. Attempted to drop it; reverted because `tsgo -b --noEmit` resolves `bun-types` for `apps/api` as `primary: false` and the global `Bun` namespace stops loading when only `apps/api` declares the type. Revisit when (a) tsgo's project-references type resolution improves, (b) we switch from `bun-types` to `@types/bun` (the modern Microsoft-maintained types), or (c) we drop `moduleResolution: bundler` for `apps/api`. In the meantime, an oxlint rule could forbid `Bun\.` and `import.meta.dir` in `apps/web/src/**` to catch any leak before it ships.
- `SPA_DIST` is resolved at module load relative to `import.meta.dir` in `apps/api/src/index.ts:6`. This breaks any future `bun build --compile` workflow that produces a single binary, because the compiled binary's `import.meta.dir` no longer points at the dist tree. Revisit when a binary-bundling story arrives; allow override via env var.
- `process.stdout.write` of the structured "listening" line in `apps/api/src/index.ts:46` is not awaited or explicitly flushed. There is a theoretical race where `app.listen` fails or the process exits before the line reaches a downstream reader (e.g. `lhci`'s `startServerReadyPattern: "listening"`). lhci is currently working per the Change Log, so defer until we see a flake.
- `check:release` runs `lhci autorun` twice (mobile, then desktop), both binding port 3000 via the same `startServerCommand`. If the first run leaves the API server alive, the second `EADDRINUSE`s. lhci historically cleans up between runs, so defer until a flake appears; if it does, switch the desktop run to a separate port or add a wait gate.

## Deferred from: code review of 1-2-backend-plumbing-logger-error-envelope-request-middleware-health-endpoint (2026-04-30)

- `apps/api/src/log.ts:23` — `JSON.stringify({ level, msg, ts, ...extra })` will throw on circular references and on BigInt-typed field values. Today's callers pass narrow types, so risk is low, but a future caller logging a request body or domain entity could crash the request path. Add a JSON.stringify replacer (WeakSet seen-tracker + BigInt → string) or wrap `emit` in try/catch with a fallback line.
- `apps/api/src/middleware/requestLogger.test.ts` — no concurrent-request test asserts that two overlapping `app.handle()` calls produce distinct, non-interleaved `requestId` values. The `WeakMap<Request, string>` design is correct for concurrency, but a regression introducing module-level mutable state would pass all current tests. Add a test that fires two `app.handle()` promises in parallel and asserts each pair of entry/exit logs shares an id distinct from the other pair.
- `apps/api/src/onError.test.ts` — no test asserts `Content-Type: application/json` on error envelope responses. `routes/health.test.ts` checks it for the happy path; error responses do not. If a future Elysia upgrade emits envelopes as `text/plain`, strict client parsers break silently.
- `apps/api/src/errors/AppError.ts` — class does not accept the standard `ErrorOptions.cause` option. Wrapping an upstream DB/network error (Stories 1.3+) loses the original stack and message. Add an optional `cause` field passed through to `super(message, { cause })`.
- `apps/api/src/middleware/requestLogger.ts:33`, `apps/api/src/onError.ts:15` — `requestLogger` and `registerOnError` return `AnyElysia`, which erases the chain's `derive`/`decorate` typing for downstream `app.use(...)` consumers. The Debug Log records this as an intentional tradeoff to avoid dragging the deeply-generic plugin type through the codebase. Track for future re-typing once Elysia's plugin type ergonomics improve.

## Deferred from: code review of 1-3-backend-persistence-tasks-table-forward-only-migrations-repository-skeleton (2026-04-30)

- `apps/api/src/storage/db.ts:14-19` — `db()` lazy singleton has no `close` / shutdown hook, no test reset, and no swap. WAL files may not checkpoint cleanly on process restart. Address when a graceful-shutdown story arrives (likely alongside Story 1.11 deployment).
- `apps/api/src/storage/migrations/runner.ts:23-55` — Concurrent migration runners against the same SQLite file have no advisory lock or `BEGIN IMMEDIATE` guard. Two booting processes can both observe `applied=∅`, both attempt `CREATE TABLE tasks`, the second fails inside its wrapping transaction. Today's deployment is single-process per container so this is theoretical; revisit if the deployment model adds workers or rolling restarts that overlap.
- `apps/api/src/index.ts:13-25` — Migration failure at boot is sticky (no retry). If the SQLite file is briefly unavailable on a network mount, /health returns 503 forever until the operator restarts the container. Consider a bounded retry-with-backoff in the boot block, or a separate "retry migrations" admin endpoint.
- `apps/api/src/storage/tasks.ts:60-72` — `INSERT OR IGNORE` swallows all constraint violations (PK, CHECK on text length, future CHECK on completed). Out-of-bounds text from an unvalidated path becomes a misleading "invariant: task missing after INSERT OR IGNORE" throw. Story 1.4's router-level validation is the primary guard; consider adding defense-in-depth at the repo boundary or translating SQLITE_CONSTRAINT_CHECK to a typed validation error.
- `apps/api/src/storage/migrations/001_create_tasks.up.sql:2` — Schema does not constrain `id` format. The repo's `ORDER BY id DESC` newest-first contract assumes UUIDv7 monotonicity; a non-UUIDv7 id injected via a future path would break ordering invisibly. Add a CHECK constraint or repository-level validation when Story 1.4 ships id-validation logic.
- `apps/api/src/storage/tasks.ts:21-29` — `list()` returns every row (no LIMIT, no pagination). Acceptable for MVP but unbounded as the table grows. Revisit when product scope adds filtering / archival.
- `apps/api/src/storage/migrations/runner.ts:32-34` — Lexical sort mis-orders 4-digit and 3-digit prefixes (`1000_y.up.sql` < `999_x.up.sql`). Spec acknowledges this; switch to numeric sort by parsed version (or fixed-width prefix) before crossing version 999.
- `apps/api/src/storage/db.ts:14-19`, `apps/api/src/storage/tasks.ts:91-103` — Lazy singletons retry the open on every call after a partial PRAGMA failure. No memoization of failure → potential reopen storm under transient errors. Consider a one-shot failure cache.
- `apps/api/src/storage/tasks.ts:60-73` — The "INSERT OR IGNORE no-op + row no longer exists" path is theoretically reachable today (manual SQL outside the repo) and becomes a real concurrency path once Story 3.1 implements `delete`. Decide between re-INSERT-without-IGNORE or documenting the no-delete invariant explicitly when Story 3.1 lands.
- `apps/api/src/storage/migrations/runner.ts:44-50` — Migration files containing explicit `BEGIN; ... COMMIT;` collide with the wrapping `db.transaction(...)` and fail with `cannot start a transaction within a transaction`. Document the convention "do not include BEGIN/COMMIT in migration files" in a comment near the runner, or strip them defensively.
- `apps/api/src/index.ts:21-25` — `new Error(String(err))` coerces non-Error throws (e.g., a thrown SQLite-error-like object whose `instanceof Error` returns false in some bun:sqlite versions) and discards the original `code`/cause. Log the raw `err` alongside the synthesized Error so SQLite-specific error codes survive into operator logs.
- `apps/api/src/storage/db.test.ts` — Default state of the readiness module (`getDbStatus()` returns `{ ready: false, error: undefined }` at module load) is not asserted in tests. Bun's test runner shares module state across test files in a single run, so once `index.test.ts`'s import-time `setDbReady()` fires, `_ready` is contaminated for all subsequent test files. The default is verifiable only via process isolation (e.g., a separate `bun test` invocation against this file alone) or by reading the source. Revisit when test isolation tooling improves, or add a guard in `index.ts` so the boot block does not run during `bun test`.
